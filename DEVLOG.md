# AI Article Illustrator 开发日志

## 起因

这个项目叫 AI Article Illustrator（全文配图生成器），核心功能是：粘贴一篇文章，AI 自动把它拆成一个个场景，为每个场景生成绘画提示词，然后调用图片模型批量出图。

整个项目是在和 Claude 的对话中一步步搭建和迭代的。下面记录了从最初的需求到最终成型的完整过程。

---

## 第一轮：拖拽上传

最开始的需求很简单——"风格参考"那个上传文件的位置，能不能直接拖拽上传？

当时项目里已经有一个隐藏的 `<input type="file">`，点按钮触发。我在外层的卡片容器上加了 `onDragEnter`、`onDragLeave`、`onDragOver`、`onDrop` 四个事件，拖文件进来的时候卡片边框高亮，文本框变成虚线提示"松开以上传文件"，松手后自动读取 `.txt` 或 `.md` 文件内容填入。

---

## 第二轮：接入火山引擎 Seedream 5.0 Lite

接着要接入火山引擎的 Seedream 5.0 Lite 图片模型。查了一圈文档发现，火山方舟的"广泛可用"模型可以直接用 model ID 调用，不需要创建接入点或购买套餐。

model ID 是 `doubao-seedream-5-0-lite`，API 格式和现有的 Seedream 4.5 完全一样（`https://ark.cn-beijing.volces.com/api/v3/images/generations`），所以后端不需要改，只在前端的模型列表里加一条就行了。

---

## 第三轮：大规模模型扩展

这是改动最大的一轮。需求变成了：

> "只要是火山的模型，我这里都可以填写。一个是语言模型，一个是图片模型。再把中国的几大模型都加上。"

### 架构改造

原来的后端只用 Anthropic SDK 调 Claude，现在需要支持 7 个不同平台的语言模型。好在除了 Claude 之外，其他平台（火山引擎、DeepSeek、通义千问、Moonshot、智谱、OpenRouter）全部兼容 OpenAI 的 `/chat/completions` 格式，所以我写了一个通用的 `callOpenAICompatible()` 函数，根据 model ID 的前缀自动路由到对应的 base URL。

图片模型这边也加了 Google Gemini 原版（就是 Nano Banana 的底层模型）。Gemini 的 API 格式比较特殊——它返回 base64 图片数据而不是 URL，所以后端把 base64 转成 data URI 返回给前端。

### Step 1 重新设计

API 设置页面从"填两个 Key"变成了"先选模型，再填 Key"的流程。文本模型和图片模型各自有一个下拉菜单，按平台分组显示。选了哪个平台的模型，Key 输入框的标签就自动变成那个平台的名字（比如选了 Doubao 就显示"火山引擎 API Key"）。

### 最终支持的模型清单

**文本模型：** Doubao 1.5 Pro/Lite/Thinking、Seed 2.0、DeepSeek V3/R1、Qwen Max/Plus/Turbo、Kimi K2.5/K2、GLM-4 Plus/Flash/Air、Claude Sonnet/Opus/Haiku、OpenRouter（一个 Key 用所有模型）

**图片模型：** Seedream 5.0 Lite/4.5/4.0/3.0/2.0、Gemini 3 Pro Image（= Nano Banana Pro）、Gemini 3.1 Flash Image（= Nano Banana 2）、Gemini 2.5 Flash Image（= Nano Banana）、以及 fal.ai 上的 Nano Banana 全系列和 Seedream

## 第四轮：风格参考保存

用户上传一次风格参考（Skill）之后希望能保存下来，下次还能用。

实现方式是在 state 里加了一个 `savedSkills` 数组，每个 Skill 有 id、name、text。保存到 localStorage，页面刷新后自动恢复。界面上在"风格参考"卡片里加了"保存 Skill"按钮，保存后显示为可点击的标签，点击加载，右边有删除按钮。

同时加了一行提示：清除浏览器数据或重置后 Skill 会丢失。

---

## 第五轮：配图模式改造

原来只有"全文配图"和"精选配图"两种模式。改成了三种：

1. **全文配图** — 改写了分段 prompt，要求"每 1-2 句话切分一个场景"，达到视频脚本级别的密度
2. **分段配图** — 用户指定配图数量，AI 把文章均匀切成指定数量的段落
3. **手动选段** — 用户在文章中滑动选中文字，点击"添加为配图段落"。选好的段落直接跳过 AI 自动分段，送去生成提示词

手动选段模式解决了用户反馈的"闪退"问题——之前试图在 Step 2 选中文字但没有对应功能，导致状态不一致。

---

## 第六轮：UI 细节打磨

这一轮处理了一堆细节：

- **去掉"提示词格式"和"目标平台"** — 用户觉得不需要
- **色彩基调加"正常"** — 表示没有特殊色调倾向
- **配图比例移到顶部** — 从风格选择器的下拉框改为显眼的按钮组，放在风格配置最上方
- **"尺寸不统一"开关** — 默认所有图片统一比例；打开后每张配图可以单独选比例
- **扩展比例** — Nano Banana / Gemini 模型额外支持 4:1、1:4、8:1、1:8
- **图片下载** — 生成的配图右上角悬浮下载按钮
- **图片尺寸修复** — Seedream 5.0 Lite 要求 ≥ 3,686,400 像素，把分辨率从 1024x1024 升级到 1920x1920+
- **侧边栏折叠** — 可以折叠为只剩图标，宽度可以拖拽调整
- **语言切换图标** — 去掉了 🌐 emoji，换成 lucide 的 Languages 图标
- **网页图标** — 加了 SVG favicon
- **示例文章** — 文章输入区新增"加载示例文章"按钮

---

## 第七轮：Google 标注 + OpenRouter

最后一轮是补充说明性文字：

- Google Gemini 图片模型的名称改为明确对照格式，比如 `Gemini 3 Pro Image (= Nano Banana Pro)`，并在描述里写明"Google 原版，fal 叫 Nano Banana Pro"
- 下拉分组标题也改为"Google 原版 (Nano Banana 原始模型)"
- 加了 OpenRouter 平台，一个 Key 就能调用多家模型

---

## 第八轮：配图比例重构

用户试用时报了一个错：Seedream 5.0 Lite 生成图片失败，提示"image size must be at least 3686400 pixels"。一看才发现，原来的尺寸映射全是小分辨率（1024x1024、1280x720），而 Seedream 5.0 Lite 要求至少 3,686,400 像素（约 1920x1920）。

把所有分辨率都升级了：1:1 改为 1920x1920，16:9 改为 2560x1440，以此类推。

同时重新设计了比例选择的交互：

- 配图比例从风格选择器的下拉框里拿出来，单独做成一排醒目的按钮组，放在风格配置最上方
- 默认是"统一比例"——所有图片用同一个比例
- 加了一个"尺寸不统一"开关，打开后每张配图的卡片上会出现独立的比例选择器
- 用 fal.ai 或 Google Gemini 模型时，自动显示扩展比例：4:1、1:4、8:1、1:8（这些极端比例是 Nano Banana 支持的）

---

## 第九轮：手动选段 + 修复闪退

用户反馈在"分段配图"模式下，想要滑动选中文章的一部分来配图，但操作后点"下一步"会闪退。

问题出在：用户期望的"选中文字配图"功能根本不存在于 Step 2，自然会出错。于是加了第三种配图模式——"手动选段"：

- 用户先粘贴文章（或加载示例）
- 粘贴后文章变成可选中的文本块，输入框消失
- 用鼠标滑动选中一段文字，下方即时显示选中内容
- 点"添加为配图段落"，段落加入列表
- 可以继续选其他段落，也可以删除已添加的
- 点"下一步"进入风格配置，这些手动选的段落会跳过 AI 自动分段，直接送去生成提示词

同时还加了示例文章——一段描写老街清晨到傍晚的散文，方便测试。文章内容以很低的透明度显示在空白输入框里，点击就会加载进去。

---

## 第十轮：图片风格参考

之前的"风格参考"只能输入文字描述，用户觉得不够直观。这一轮加了图片风格参考：

- "风格参考"区域新增"上传参考图"按钮
- 用户上传一张图片后，前端先把图片缩放到最大 512px（用 Canvas API），然后自动调用后端的 `/api/analyze-style` 接口
- 后端把图片发给用户选的语言模型（Claude 用 Anthropic SDK 的 vision 格式，其他平台用 OpenAI-compatible 的 image_url 格式），让 AI 分析图片的艺术风格、色彩方案、构图特点、质感纹理、光影效果、情绪氛围、细节程度
- 分析结果自动填入文字描述框，用户可以继续编辑微调
- 图片缩略图 + 文字描述可以一起保存为"风格"，下次直接加载
- 拖拽也支持了图片——拖一张图进来，自动识别是图片还是文本文件，走不同的处理流程

---

## 第十一轮：下载与界面优化

这一轮处理了一堆实际使用中的问题：

**下载全部图片：** 在结果页工具栏加了"下载全部图片"按钮。一开始用前端 `fetch` 直接下载图片 URL，结果因为跨域（CORS）完全没反应。解决方案是加了一个 `/api/proxy-image` 后端代理，服务端去请求图片再转发给浏览器，绕过了跨域限制。单张下载也改成走这个代理。

**工具栏两行布局：** 原来所有按钮挤在一行，模型名字长的时候按钮会变形。改成两行：
- 第一行放主操作：生成全部配图、下载全部图片
- 第二行放辅助操作：选文配图、复制提示词、导出 MD、导出 JSON，用小字 + 竖线分隔

**右上角关注入口：** 页面右上角加了一个"关注@阿真Irene"按钮，鼠标悬停弹出微信二维码卡片，下方有一行文案"AI 工具 · 前沿资讯 · 有趣玩法 都在这里"。

**清理命名：** 把所有界面上的"Skill"字样统一改为"风格"——"已保存的风格"、"保存风格"、"风格名称"，避免引起误解。

## 技术栈

- Next.js 16 + React 19
- Tailwind CSS v4 + Radix UI
- Lucide React 图标
- Anthropic SDK + OpenAI-compatible API 通用调用
- React Context + useReducer 状态管理
- TypeScript 全栈类型安全

## 总结

整个项目从一个简单的"拖拽上传"需求开始，在持续对话中逐步演化成了一个支持 7 个 AI 平台、20+ 模型、3 种配图模式的完整工具。每一轮迭代都是在用户实际使用中发现问题、提出新需求，然后即时实现的。没有预先设计文档，没有需求评审，就是不断对话、不断试用、不断修改。最终的成品比一开始任何人预想的都要丰富得多。
